JWT vs 세션 관리(로그아웃 처리)

JWT:
무상태성: 서버는 클라이언트의 상태를 저장하지 않아 확장성이 뛰어남. 하지만 토큰은 발급 후 만료 시간이 지나야만 무효화
로그아웃 처리 어려움: JWT는 서버에서 제어권이 없으므로, 로그아웃 시 Redis와 같은 별도의 저장소를 활용해 토큰 블랙리스트를 구현하거나 만료 시간을 짧게 설정해야함
보안 이슈: 민감한 정보를 페이로드에 포함하지 않아야 하며, HTTPS를 통해 안전하게 전송해야 함
세션관리:
상태 저장: 서버가 클라이언트의 세션 정보를 유지하며, 로그아웃 시 세션을 무효화할 수 있음
보안성: 세션 ID는 서버에서만 관리되므로 탈취 위험이 낮음
확장성 문제: 분산 시스템에서는 모든 서버가 동일한 세션 저장소를 공유해야 하므로 관리가 복잡함

JWT 클레임 활용

* JWT 클레임: name / value 쌍으로 이뤄진 payload 부분의 조각 *
Registered Claim: 토큰 정보를 표현하기 위해 미리 정해진 종류의 데이터
ex) iss발급자, sub주체, aud청중, exp만료 시간)
Public Claim: 사용자 정의 클레임으로 공개용 정보를 위해 사용. 충돌방지를 위해 클레임 이름을URI 형태로 사용
ex)
JSON
복사
{ "https://altair.com/jwt_claims/is_admin": true }
​
Private Claim: 사용자 정의 클레임으로 서버와 클라이언트 사이에 임의로 지정한 정보를 저장
ex)
JSON
복사
{ "access_token": access }
​
특정 요구사항에 맞춰 사용자 역할, 권한, 구독 상태 등을 추가 가능 그러나, 과도한 정보는 토큰 크기를 증가시켜 네트워크 비용 부가될 수 있음

JWT 확장성 및 비용

확장성 • JWT는 상태 비저장 특성 덕분에 서버 간 상태 공유가 필요 없어 마이크로서비스 아키텍처에서 유용 • 새로운 서버 추가 시, 별도의 세션 동기화 작업 없이 독립적으로 토큰 검증이 가능하므로 수평적 확장이 용이
비용 • 네트워크 비용: JWT 크기가 커지면 매 요청마다 전송되는 데이터량이 증가해 네트워크 비용이 높아질 수 있음 • 유지보수 비용: 각 서비스에서 독립적으로 검증 로직을 구현해야 하므로 초기 개발 및 유지보수 비용이 증가할 수 있음
