MSA 팀플 정리 (2)

1. Spring Cloud Config 자동 리로드

Spring Cloud Config는 설정 파일을 Git 저장소에서 관리하고, 각 마이크로서비스가 해당 설정을 중앙에서 가져오는 방식
설정 변경 시 서비스를 재시작하지 않고 반영하려면 spring-cloud-bus 를 활용해 /actuator/bus-refresh 요청을 사용
Bus는 메시지 브로커(RabbitMQ, Kafka)를 통해 변경 이벤트를 전체 서비스에 전파

2. Spring Cloud Bus의 기능

설정 버전 동기화: 서비스 간 설정 버전을 동일하게 유지하여, 설정 불일치에 따른 장애를 방지 가능
설정 롤백 처리: Git에서 이전 커밋으로 되돌린 후 /bus-refresh 요청으로 전 서비스에 설정을 일괄 재적용 가능
이벤트 기반 배포 자동화: Git 저장소의 설정 파일 변경 시 Webhook으로 Jenkins 등 CI/CD 파이프라인을 트리거하고, 배포 후 /actuator/bus-refresh까지 자동으로 호출되면 사람의 개입 없이 설정 변경 → 배포 → 반영까지 이어지는 자동화 가능! 이는 대규모 MSA 환경에서 설정 일관성과 운영 효율성을 동시에 확보할 수 있는 방식
Kafka 연동 시 메시지 추적 가능: 설정 변경이 언제, 어디에 반영되었는지를 Kafka 로그를 통해 추적 가능

실무 적용 시 유의 사항

메시지 브로커(RabbitMQ 또는 Kafka) 인프라 구성이 필요하며, 운영 환경에서는 서비스가 언제나 이용 가능하도록 유지할 수 있는 능력이 필요
설정 보안이 중요한 경우 암호화 및 접근 제어 정책을 반드시 적용해야 함

3. Service Mesh

Service Mesh는 마이크로서비스 간의 통신을 프록시(Proxy) 기반으로 제어하는 아키텍처
기존에는 서비스 코드에서 직접 처리하던 인증, 암호화, 로깅, 장애 복구 등을 서비스 외부의 인프라 레이어에서 처리한
대표적인 Service Mesh 솔루션은 Istio, Linkerd 등이 있으며, 일반적으로 사이드카 프록시(예: Envoy) 구조를 따름

동작 구조

각 마이크로서비스 옆에 사이드카 프록시 컨테이너가 함께 배포
서비스 간 통신은 직접 연결되지 않고, 프록시 → 프록시 → 대상 서비스로 전달
프록시는 통신 중 다음과 같은 기능을 자동으로 수행:
기능
설명
암호화(mTLS) 자동 적용
서비스 간 트래픽을 자동으로 암호화하고 인증 수행
인증/인가 제어
JWT 기반 사용자 인증을 포함한 정책 설정 가능
트래픽 관리
특정 버전에만 트래픽 전송, 비율 기반 배포(Canary), 장애 시 자동 리트라이 등
관찰 가능성 향상
메트릭, 로그, 트레이싱 정보를 중앙 수집 시스템(Grafana, Prometheus 등)과 통합 가능

장점

보안: 인증 및 암호화를 자동화하여 보안 수준을 높일 수 있음
운영 효율: 배포 전략, 장애 대응 등을 인프라 수준에서 제어 가능
ex) 10%만 새로운 버전으로 배포하고 문제 없으면 100%로 배포하는 Canary 배포 전략을 설정 파일만 수정해서 이용 가능
코드 독립성: 서비스 코드에 네트워크 관련 로직이 필요 없음

단점

초기에 설정하기 어렵고 배우기도 어려움
서비스 옆에 사이드가 프록시가 하나씩 붙어서, 서비스 하나당 컨테이너가 2개가 됨, 메모리와 CPU의 사용이 거의 2배가 됨
잘못된 정책 설정 시 전체 트래픽 장애로 이어질 수 있음

실무 적용 시 고려 사항

Kubernetes 환경과 함께 사용하는 것이 일반적이며, Helm을 통한 설치 및 관리가 가능
점진적 도입 전략이 필요하며, 초기에는 일부 서비스에만 적용해 보는 방식이 안정적
모니터링 도구(Grafana, Kiali, Prometheus 등)와 함께 사용하는 것이 효과적
